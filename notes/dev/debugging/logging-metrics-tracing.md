# 로그와 메트릭과 트레이싱

## 한 줄 요약

로그는 맥락을 설명하고, 메트릭은 추세를 보여 주고, 트레이싱은 요청 단위 흐름을 연결해 준다.

## 언제 쓰나

- 운영 이슈를 어떤 신호로 볼지 정리하고 싶을 때
- 장애 원인이 코드, 인프라, 외부 의존성 중 어디에 있는지 가늠할 때
- 한 요청이 여러 서비스와 큐를 거치는 시스템을 추적할 때

## 핵심 개념

- 로그는 상세하지만 질의와 집계가 느릴 수 있다.
- 메트릭은 싸고 빠르지만 원인 설명력은 낮다.
- 트레이싱은 분산 시스템 병목을 찾기 좋지만 모든 질문에 답하지는 못한다.

## 실전 예시

- 에러율과 레이턴시는 메트릭으로 먼저 감시한다.
- 특정 실패 요청의 입력과 상태 전이는 구조화 로그로 본다.
- 서비스 간 병목은 트레이스로 구간 시간을 비교한다.

## 주의할 점

- 로그에 민감 정보나 과한 페이로드를 남기지 않는다.
- 메트릭 이름과 태그가 일관되지 않으면 운영 판단이 흔들린다.
- 트레이싱이 있다고 로그 설계를 건너뛰면 실제 디버깅 속도가 느려진다.
