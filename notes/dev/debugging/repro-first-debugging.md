# 재현 우선 디버깅

## 한 줄 요약

좋은 디버깅은 원인을 바로 추측하는 것보다 문제를 재현 가능한 최소 형태로 줄이는 데서 시작한다.

## 언제 쓰나

- 버그가 간헐적으로 발생해서 원인을 특정하기 어려울 때
- 운영 장애를 로컬이나 스테이징에서 다시 만들어 보고 싶을 때
- 여러 가설이 동시에 떠올라 어디서부터 봐야 할지 막막할 때

## 핵심 개념

- 재현 조건을 입력, 환경, 타이밍으로 쪼개면 문제를 더 좁힐 수 있다.
- 최소 재현 사례가 생기면 로그, 메트릭, 트레이싱을 어디에 붙일지 결정하기 쉬워진다.
- 재현이 안 되는 상태에서 코드만 훑으면 설명은 많아지고 검증은 약해진다.

## 실전 예시

- 동일 요청을 반복해 실패율이 올라가는지 본다.
- 데이터 샘플을 줄여 특정 레코드나 특정 상태에서만 깨지는지 본다.
- 비동기 코드라면 타임아웃, 취소, 재시도 조건을 분리해 본다.

## 주의할 점

- "재현 안 됨"을 문제 없음으로 해석하면 안 된다.
- 증상 설명과 재현 절차를 한 문서에 남겨야 다음 사람이 이어받기 쉽다.
- 계측을 추가하더라도 질문이 모호하면 로그만 늘고 답은 늘지 않는다.
