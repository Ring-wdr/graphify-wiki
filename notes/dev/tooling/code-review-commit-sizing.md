# 코드 리뷰를 위한 커밋 크기

## 한 줄 요약

좋은 커밋은 작아서 읽기 쉬운 것이 아니라, 하나의 의도를 끝까지 설명할 수 있을 만큼만 작아야 한다.

## 언제 쓰나

- 리뷰 속도가 느리거나 피드백 품질이 낮을 때
- 한 PR에 구조 변경, 리팩터링, 기능 추가가 섞일 때
- 체리픽, 리버트, 충돌 해결이 자주 어려울 때

## 핵심 개념

- 커밋 단위는 파일 수보다 변경 의도 기준으로 나누는 것이 중요하다.
- 리팩터링과 동작 변경을 분리하면 리뷰와 회귀 분석이 쉬워진다.
- 작은 커밋은 체리픽, 리버트, 블레임 추적의 비용도 함께 낮춘다.

## 실전 예시

- 이름 변경만 있는 커밋과 동작 수정 커밋을 나눈다.
- 허브 문서 개편과 원자 노트 내용 추가를 분리한다.
- 큰 기능도 "스키마", "동작", "정리"처럼 의도별로 쪼갠다.

## 주의할 점

- 너무 잘게 쪼개서 맥락이 사라지면 오히려 리뷰가 어려워진다.
- 임시 디버깅 코드와 실제 수정 코드를 같은 커밋에 넣지 않는다.
- 커밋이 작더라도 제목이 모호하면 히스토리 가치는 떨어진다.
