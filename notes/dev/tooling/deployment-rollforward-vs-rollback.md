# 롤포워드와 롤백

## 한 줄 요약

배포 사고 대응에서는 무조건 되돌리는 것보다 현재 상태를 기준으로 더 안전한 방향을 고르는 판단이 중요하다.

## 언제 쓰나

- 새 배포 이후 오류율이나 레이턴시가 급격히 나빠졌을 때
- 데이터 스키마나 외부 연동 때문에 단순 롤백이 위험할 때
- 장애 대응 절차를 미리 정리하고 싶을 때

## 핵심 개념

- 롤백은 빠르지만 데이터와 호환성이 꼬이면 더 위험할 수 있다.
- 롤포워드는 수정 배포가 필요하지만 현재 상태를 보존하며 복구할 수 있다.
- 대응 판단에는 계측 품질, 변경 범위, 데이터 호환성, 배포 속도가 함께 들어간다.

## 실전 예시

- 단순 UI 회귀는 롤백이 빠를 수 있다.
- 이미 마이그레이션이 적용된 쓰기 경로는 롤포워드가 더 안전할 수 있다.
- 장애 중에는 원인 탐색과 함께 "다음 안전 상태"를 먼저 정의한다.

## 주의할 점

- 롤백 경로를 평소에 검증하지 않으면 위기 때 더 큰 사고가 난다.
- 수정 배포를 택하더라도 관측 지표와 중단 기준이 명확해야 한다.
- 핫픽스를 급히 만들수록 커밋 단위와 설명을 더 작게 유지해야 한다.
